## Правила кодирования документов

Система организации документации является **иерархической**. Каждый документ имеет строгое правило именования:

> <код>-<название документа>

Где `<код>` - идентификатор в справочной системе документации для эффективного поиска и группировки документов в единную иерархию.
Как правило нименования пишутся на русском языке так, как основной контингент разработчиков и пользователей русскоязычный.

Пример:
> `01-руководство-по-использованию-документации.md`

## Правила оформления документов

Как правило у каждого документа есть **метаданные** или же **атрибуты**, которые позволяют идентифицировать автора, дату создания и ответственного для конкретного документа. Также каждый документ должен иметь **оглавление** для удобной навигации и эффективной итерации по документу. Таким образом обязательная структура документа выглядит следующим образом:

```markdown
    # Название документа

    > **Автор:** <Имя автора> <br> **Ответственный:** <Имя ответственного> <br> **Дата создания:** <Дата создания в формате "день месяц год">

    ## Оглавление
    1. [Первый раздел](#первый-раздел)
    2. [Второй раздел](#второй-раздел)

    ## Первый раздел
        ...

    ## Второй раздел
        ...
```

## Версионирование документов

Так как документы могут изменяться со временем, необоходимо следить за изменениями. Для этого используется **система управлениями версиями (VCS)**, в данном случае используется **Git**, а конкретно реализация в системе управления проектами **GitHub**. Для того чтобы чётко понимать какое изменение произошло необходимо указывать изменения в **commit**.

> **Commit (коммит)** - это зафиксированный **«снимок» (snapshot)** состояния файлов проекта в определенный момент времени. Он работает как контрольная точка или «сохранение» в игре, записывая изменения (добавление, удаление, редактирование файлов) из **области подготовленных файлов (staging area)** в историю репозитория.
